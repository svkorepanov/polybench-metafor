\textbf{Optimizing compilers} described in section \ref{sec:2_compiler_opt} enhance performance by transforming code into efficient machine code, but they face limitations such as underutilizing hardware features and opaque optimization processes.

\textbf{Compiler directives and pragmas} in section \ref{sec:2_pragmas} provide a mechanism for programmers to guide compiler optimizations through annotations embedded in source code often called pragmas. Directives such as OpenMP and OpenACC have become standard tools for expressing parallelism and optimization hints in HPC applications, particularly for targeting multicore processors and accelerators ~\parencite{chargueraud_optitrust_2022_0014}.

\textbf{Domain-Specific Languages} (DSLs) in section \ref{sec:2_dsl} provide a different way to enhance performance by offering high-level representations designed for particular problem areas. In High-Performance Computing (HPC), DSLs allow developers to articulate computational algorithms in a way that distinguishes the algorithm's purpose from implementation specifics, enabling compilers or code generators to automatically perform optimizations tailored to the domain. However, using DSLs requires programmers to rework current applications in the new language, which poses a major obstacle for adopting legacy HPC software. As noted by ~\textcite{hyperf_2025_0013} in their examination of the HYPERF framework, numerous HPC applications embody years of development and expertise captured in languages such as Fortran. Rewriting these applications in a DSL is often not feasible due to the time commitment needed and the danger of introducing mistakes. Another hurdle with DSLs is the time it takes to learn a new language and its programming paradigm. While DSLs can offer strong abstractions, they demand that programmers gain knowledge in domain-specific ideas and practices ~\parencite{rival_loopy_2016_0008}. This effort might be worthwhile for developing new applications but creates a considerable obstacle for the upkeep and optimization of existing software. Lastly, another limitation of DSLs is their inflexibility in complex optimization situations ~\parencite{bispo_clava_2020_0001}. When a developer needs a slightly more tailored solution to their problem, it may not be feasible to achieve the needed optimizations within the DSL's limits, requiring a return to manual coding methods.

\textbf{Code transformation via rewrite tools} in section \ref{sec:2_rewrite_tools}  represent an approach that operates directly on source code representations, applying pattern-matching and rewriting rules to modify program structure. This approach offers flexibility in expressing transformations and can be applied to existing codebases without requiring complete rewrites. However, it is limited to numbers of patterns a tool is able to recognize. And it is hard to apply second round if transformation on already transformed code.

\textbf{Scriptable Source to Source Compilers} in section \ref{sec:2_scripts} have a script written in any programming language that contains instruction on how and in what order to apply transformation to the original source code. The key advantages of this approach are control over the order in which transformation are applied, ability to share transformation scripts. These tools encourage to apply several rounds of transformation leading to achieving better performance. And the most important, an optimized for performance and hard to read source code is kept separate from the high level source code that describe domain algorithms, which leads to easier maintenance and overall evolution of the code base. Being able to apply different transformation scripts is also makes it easier to experiment with various optimization strategies and adapt to changing hardware capabilities, ultimately enhancing the performance of HPC applications.

\section{Compiler directives, pragmas or annotations} \label{sec:2_pragmas}

Compiler directives provide a mechanism for programmers 
to guide optimizations through annotations 
embedded in source code often called pragmas. Directives such as OpenMP \footnote{\url{https://www.openmp.org/}} and 
OpenACC~\footnote{\url{https://www.openacc.org/}}
have become standard tools for expressing parallelism and optimization 
hints in HPC applications, particularly for targeting 
multicore processors and accelerators.

\subsection[Orio]{Orio~\parencite{orio_2009_0007}}

Orio is an extensible annotation-based empirical performance tuning 
system developed by Albert Hartono, Boyana Norris, and P. Sadayappan, 
affiliated with Ohio State University and Argonne National Laboratory. Its primary mission is to enhance both 
performance and productivity by allowing software developers to embed annotations as structured comments 
directly into their source code. These annotations trigger low-level performance optimizations on 
specific code fragments. Orio aims to maximize performance tuning opportunities by supporting both 
architecture-independent and architecture-specific code optimizations, and it is designed to be portable, 
extensible, and automatable, allowing for integration with other source transformation tools. 
The system was developed as part of research supported by the U.S. Dept. of Energy under Contract DE-AC02-06CH11357.

Orio is capable of generating numerous tuned versions of a given operation and empirically 
evaluating them to select the best-performing one for production use. 
It can apply various source-to-source transformations, such as loop 
unrolling, loop tiling, and loop permutation, to improve performance. 
Beyond these, Orio supports a range of optimizations including memory alignment,
loop unroll/jamming, scalar replacement, register tiling, loop-bound replacement, 
array copy optimization, and multicore parallelization using OpenMP.
It also facilitates architecture-specific optimizations, such as generating 
calls to SIMD intrinsics for Intel and Blue Gene/P architectures. Orio can 
be used as an automatic performance-tuning tool, producing optimized code 
variants for different performance parameter values, executing them,
and evaluating their performance to identify the best version. 
This empirical tuning process can be guided by search heuristics 
like random search, Nelder-Mead simplex method, and simulated annealing 
to navigate large search spaces efficiently. 

The workflow with Orio begins when the user provides annotated C source code as input. 
The annotations are embedded as structured comments, 
starting with \lstinline{/*@} and ending with \lstinline{@*/}, and they define regions for transformation. 
These annotations include a leader annotation specifying the code transformation module 
(e.g., \lstinline{Loop} for loop transformations or \lstinline{BGP_Align} for memory alignment) 
and performance-tuning directives. 
Orio scans the input, extracts annotation regions, 
and processes them using corresponding transformation modules.
The code generator then produces the final C code with the specified optimizations.
Orio works primarily with C code, and its annotation language, which describes transformations,
is embedded within C/C++ comments. While initially focused on C, 
future work includes providing support for annotating and generating Fortran code.
The system can also be integrated with external source transformation tools 
like PLuTo, where PLuTo performs polyhedral transformations and 
Orio applies complementary syntactic transformations.

In essence, Orio serves as a flexible and powerful framework 
for empirically optimizing high-performance computing applications 
by decoupling optimization strategies from the core algorithm 
through an annotation-based approach, supporting a variety of transformations and architectures.

\subsection[X Language]{X Language~\parencite{x_language_2006_0012}} \label{sec:se_xlanguage}

The X Language is an annotation-based tool designed to facilitate 
the creation of high-performance code, particularly for matrix-matrix 
multiplication (MMM) routines. It was developed by researchers including 
Sebastien Donadio, James Brodman, Thomas Roeder, Kamen Yotov, Denis Barthou,
Albert Cohen, María Jesús Garzarán, David Padua, and Keshav Pingali,
with contributions from institutions like BULL SA, University of
Versailles St-Quentin-en-Yvelines, INRIA Futurs, University of
Illinois at Urbana-Champaign, and Cornell University.
The primary mission of X is to provide a framework for programmers and
library generators to represent parameterized programs compactly
and naturally, enabling a powerful optimization framework for high-performance computing.
This approach aims to address the increasing complexity of processors and
the difficulty compilers face in delivering optimal performance by allowing
empirical search to find the best-performing program versions.

The workflow with the X Language involves translating an annotated C program in two main steps. First, a frontend parses the annotated C program, builds an abstract syntax
tree, identifies loops and transformations specified by X language directives,
and then rewrites marked loops as library calls. Transformation directives
are also translated into library calls for appropriate transformations on these loops.
After all annotations are processed, the remaining code is transformed using
a multistage language, which then produces the final optimized C code.
This generated C code is then executed, reading a separate file that
describes optimizations, performing them, and producing the final optimized C code.
The macro language is used to manipulate code expressions and write optimizations,
with partial evaluation of expressions containing variables and constants.
The X Language itself uses \lstinline{#pragmas} to name loops or code sections and to
specify transformations, such as \lstinline{unroll} and \lstinline{stripmine}. 
It also supports array scalarization, which involves loading array elements into scalar variables.

The X Language is designed to work with C++ programs, particularly for high-performance computing, and can be extended to represent C code and common C++ dialects.
The transformations in X are described using pattern rewriting rules and macro code.
For instance, a \lstinline{stripmine} transformation can be defined using
a code template followed by the resulting form after modification.
This allows for the generation of various program versions, 
with parameters like tile size and unrolling degree being 
determined through empirical search. The tool's infrastructure
is built to be compiler-independent, focusing on a fully typed
Internal Program Representation (IPR) that represents C++ code and allows
for semantic-based analysis and transformations.

\subsection[Scout]{Scout~\parencite{scout_2012_0017}}

Scout is a source-to-source transformation tool specifically designed for SIMD (Single Instruction, Multiple Data) optimizations, primarily for vectorizing C source code.
It was developed by researchers including Olaf Krzikalla, Kim Feldhoff,
Ralph Müller-Pfefferkorn, and Wolfgang E. Nagel from Technische Universität, Dresden, Germany.
The tool's main mission is to provide maximum flexibility in vectorizing a wide range of loop
constructs and targeting various modern SIMD architectures, such as SSE and AVX.
Scout was developed in the context of the HI-CFD project,
which aimed to comfortably vectorize C loops for various HPC platforms
with different instruction sets and compilers. It functions as a configurable
source-to-source transformator to automatically vectorize C code,
addressing the challenges compilers face in optimizing complex loops for SIMD units.

Scout's capabilities revolve around its ability to transform C source code to exploit parallel SIMD units, producing C source code with vectorized loops augmented by SIMD intrinsics.
It is designed to be a universal vectorizer and an industrial-strength vectorizing preprocessor.
The tool supports vectorization of various data types, such as \lstinline{float} and
\lstinline{double}, simultaneously within a single loop by using an unroll-and-jam approach.
This method allows for partial unrolling of statements based on vector
size and merging vectorizeable statements, while unvectorizeable statements (e.g., `if`-statements) remain unrolled. Scout also handles array scalarization, where array
elements are loaded into scalar variables . Furthermore, it can perform partial
vectorization for loops with indirect indexing or dependencies,
allowing the computation part to be vectorized while accumulation remains serial.
Before vectorization, Scout simplifies the loop body through function inlining,
loop unswitching, and unrolling of inner loops.

The workflow with Scout involves a semi-automatic vectorization process where the programmer annotates loops using \lstinline{#pragma} directives. Specifically,
\lstinline{#pragma scout loop vectorize} placed before a \lstinline{for}
statement triggers its vectorization. Internally, Scout uses the \lstinline{clang} parser
to transform the C source code into an Abstract Syntax Tree (AST).
Optimizations and vectorization are then performed on this AST,
which is subsequently rewritten back into C code.
The tool's configurability is a central requirement, achieved by supplying
a configuration file written in pure C++ that describes the properties of the target SIMD platform.
This configuration allows defining how intrinsics are wrapped in string literals,
making the configuration valid C++ even if target SIMD headers are unavailable.
The transformations themselves are described using pattern rewriting rules and macro code,
enabling flexible generation of various program versions and empirical
search for optimal parameters. Scout is compiler-independent and
focuses on a fully typed Internal Program Representation (IPR) for C++ code analysis and transformation.

\section{Domain-Specific Languages} \label{sec:2_dsl}

\subsection[Halide]{Halide~\parencite{halide_2012_0021}}

Halide is a domain-specific language (DSL) developed by Jonathan Ragan-Kelley,
Andrew Adams, Sylvain Paris, Marc Levoy, Saman Amarasinghe, and Frédo Durand,
affiliated with institutions like MIT CSAIL, Adobe, and Stanford University.
Its primary mission is to decouple the algorithm from its schedule in image
processing pipelines, allowing for high-performance implementations without
sacrificing code clarity, readability, or portability. This separation simplifies
algorithm specification by treating images and intermediate buffers as functions
over an infinite integer domain, removing the need for explicit storage or boundary conditions.
Halide was designed to enable programmers to efficiently explore different
optimization strategies without altering the algorithmic code, thereby
reducing the "programmer pain" and code complexity associated with achieving
high performance in image processing. The research was supported by grants f
rom the Quanta T-Party, NSF, DOE, and gifts from Cognex and Adobe.

Halide is highly capable of generating efficient machine code for various architectures,
including ARM, x86, and GPUs, targeting SIMD units, multiple cores, and complex memory hierarchies.
It supports a wide range of image processing algorithms, such as camera raw pipelines,
bilateral grids, fast local Laplacian filtering, and image segmentation.
The language represents algorithms functionally, where values are pure functions
from coordinates to values, rather than mutable arrays. This functional
representation, along with the separation of algorithm and schedule,
allows for flexible analysis and transformation during compilation.
Halide can express common image processing operations like convolutions,
histograms, image pyramids, and complex stencils. It also supports reductions,
which are defined recursively and consist of an initial value function
and a recursive reduction function . The framework allows for various scheduling
choices, including tiling, fusion, recomputation versus storage, vectorization, and parallelism.

The workflow with Halide involves the programmer first defining the imaging
pipeline using Halide functions, which represent the intrinsic algorithm.
This algorithm is expressed in a C++ embedded language, allowing for JIT
compilation or static compilation to an object file and header. After specifying
the algorithm, the programmer separately defines a partial schedule for each
function, which dictates how the algorithm should be executed on a particular machine.
The compiler then removes syntactic sugar, generates a complete schedule, and
lowers the pipeline into an imperative representation. Bounds inference is
performed to compute the bounds of each loop and the size of intermediate buffers.
The representation is then further lowered to LLVM IR for compilation to machine code.
Halide works with C++ as its host language, and the algorithms and schedules are written
using the Halide language, which is embedded within C++. This allows C++ functions
to build pipelines and construct repetitive substructures conveniently.

In essence, Halide provides a powerful abstraction for image processing by
separating the 'what' (algorithm) from the 'how' (schedule), enabling developers
to achieve state-of-the-art performance across diverse hardware platforms with
significantly more concise and maintainable code.

\subsection[Tiramisu]{Tiramisu~\parencite{tiramisu_2019_0015}}

Tiramisu is a polyhedral framework designed to generate high-performance code for
various platforms, including multicore CPUs, GPUs, and distributed machines.
It was developed by researchers including Riyadh Baghdadi, Jessica Ray,
Malek Ben Romdhane, Emanuele Del Sozzo, Abdurrahman Akkas, Yunming Zhang,
Patricia Suriana, Shoaib Kamil, and Saman Amarasinghe, affiliated with institutions
like MIT, Google, Politecnico di Milano, and Adobe. Tiramisu's mission is to bridge
the gap between compiler-generated code and hand-tuned libraries by providing
fine-grained control over optimizations through a rich scheduling language.
It is particularly well-suited for image processing, deep learning, linear algebra,
tensor operations, and stencil computations. This work builds upon a preliminary
version published on ArXiv  and has received support from various grants,
including ADA Research Center, a JUMP Center co-sponsored by SRC and DARPA.

Tiramisu is capable of generating highly optimized code for complex data-parallel
algorithms by applying a wide range of transformations. It supports common affine loop
transformations, such as tiling, splitting, and shifting, as well as mapping
loop levels to hardware for parallelization, vectorization,
and GPU block/thread dimensions. Beyond traditional loop transformations,
Tiramisu introduces novel commands for explicit communication and synchronization, 
and for mapping buffers to different memory hierarchies, enabling it to
target distributed systems effectively. For instance, it can automatically
compute data amounts for caching and copying, and insert necessary synchronization.
Tiramisu can handle non-rectangular iteration spaces and programs with
cyclic data-flow graphs, unlike some other frameworks. It can generate efficient
code that matches or outperforms state-of-the-art compilers and hand-tuned
libraries across different hardware architectures. For example, it can achieve
up to 2.3x speedup over Intel MKL for VGG neural networks and significantly
outperform Halide in certain image processing benchmarks due to its ability
to apply loop fusion and handle complex data access patterns.

The workflow with Tiramisu involves the user writing the pure algorithm and
then providing a set of scheduling commands . Tiramisu is implemented as
a domain-specific language (DSL) embedded in C++, providing a C++ API for
users to write architecture-independent algorithms and scheduling commands.
The framework uses a four-level intermediate representation (IR) that separates
the algorithm from loop transformations, data layouts, and communication,
simplifying the targeting of multiple hardware architectures.
The scheduling commands, which are written using the Tiramisu language embedded
in C++, guide the code generation process. Tiramisu uses integer sets and maps,
based on the Integer Set Library (ISL), to represent iteration domains,
memory accesses, and transformations across its IR layers. The final output
is optimized backend code, including LLVM IR for multicore CPUs and distributed
machines, and LLVM IR + CUDA for GPUs.

In summary, Tiramisu is a polyhedral compiler that leverages a C++ embedded DSL
and a multi-layered IR to provide fine-grained control over optimizations
for HPC applications. It excels at generating high-performance code for diverse
architectures, supporting complex transformations, and handling challenges in image
processing, deep learning, and linear algebra by explicitly separating algorithmic
concerns from scheduling details.

\section{Code Transformations via Rewrite Tools} \label{sec:2_rewrite_tools}

\subsection[EPOD]{EPOD~\parencite{epod_0019}}

EPOD (Extendable Pattern-Oriented Optimization Directives) is a framework developed
by Huimin Cui, Jingling Xue, Lei Wang, Yang Yang, Xiaobing Feng, and Dongrui Fan
from institutions like the Chinese Academy of Sciences and the University of New South Wales.
Its mission is to enable algorithm-specific optimizations to be encapsulated into
"optimization patterns" and expressed through preprocessor directives, thereby
integrating domain experts' experience and knowledge into general-purpose compilers.
The core idea is to narrow the performance gap between compiler-generated and
hand-tuned code by exploiting algorithm-specific, semantic-specific optimizations
within a flexible compiler framework . EPOD provides an Optimization Programming Interface (OPI)
for compiler developers to define new optimization schemes and integrate new patterns flexibly.
This work extends a paper presented at the International Symposium on Code Generation and
Optimization (CGO'11) and is partially supported by grants from the
Chinese National Basic Research Grant, NSFC, and the Australian Research Council.

EPOD is capable of applying a wide range of transformations, particularly
those that are algorithm-specific and often missed by traditional compilers.
It supports patterns like \lstinline{stencil}, \lstinline{relaxed-stencil},
\lstinline{dense-mm} (for matrix multiplication), \lstinline{dmdarray}
(for dynamically allocated arrays), and \lstinline{compressed-array}.
These patterns encompass various loop transformations, data layout
optimizations, and even fine-grain synchronization. For instance, \lstinline{dense-mm}
can apply loop fission and peeling after tiling to optimize matrix multiplication. 
The \lstinline{dmdarray} pattern automatically changes the underlying data layout
of dynamically allocated arrays from discontinuous to continuous, and transforms
multidimensional subscripting expressions into one-dimensional ones, which
is beyond typical compiler capabilities. EPOD can also handle optimizations
that involve relaxing data dependencies, such as in \lstinline{relaxed-stencil},
which standard compilers cannot usually apply without human intervention.

The workflow with EPOD begins with programmers annotating their source code
using pattern-oriented directives, known as EPOD pragmas.
These pragmas conform to C and C++ standards for language directives and
include parameters specifying the pattern and its associated clauses.
The EPOD framework acts as a source-to-source translator, taking sequential
C/Fortran programs as input . The preprocessor checks if the pragma-controlled
code region satisfies the conditions for the pattern, normalizes loop nests and
labels them, and analyzes data access patterns to expose parameters for the EPOD script.
An EPOD script, written in a scripting language (discussed later), defines
 the specific optimization scheme for a pattern via the OPI.
 This script can build more than one optimization sequence, using keywords
 like \lstinline{alternative} to facilitate performance tuning.
 After transformations, the modified code is translated back to C source code
 using the \lstinline{whirl2c} routine in Open64. EPOD is designed to work with
 C and Fortran languages. The transformations themselves are described using 
 an EPOD script, which leverages the OPI to invoke individual optimization components.
 The framework is built on top of the Open64 infrastructure and uses polyhedral transformations (based on URUK) and traditional optimization pools.

In summary, EPOD is a sophisticated source-to-source transformation framework that
integrates domain-specific optimization knowledge into compilers through pattern-oriented directives.
It handles a variety of languages, primarily C and Fortran, and uses its own
scripting mechanism to define and apply complex, algorithm-specific transformations,
often leveraging the Open64 and polyhedral frameworks to achieve high performance
in HPC applications.

\section{Transformation Scripts} \label{sec:2_scripts}

\subsection[OptiTrust]{OptiTrust~\parencite{chargueraud_optitrust_2022_0014}}

OptiTrust is an interactive framework for source-to-source transformations
developed by Arthur Charguéraud, Begatim Bytyqi, Damien Rouhling, and Yann Barsamian.
Its primary mission is to enable the development of high-performance
C code by systematically applying optimization steps through transformation scripts,
thereby making the process of code optimization more manageable, reviewable,
and maintainable than manual methods.

OptiTrust is designed to facilitate the development of high-performance C
code through a series of source-to-source transformations.
It is particularly effective in optimizing complex numerical simulation codes,
such as Particle-In-Cell (PIC) simulations, by starting from a direct, unoptimized
version of an algorithm and applying state-of-the-art optimization patterns.
The framework allows programmers to interactively visualize textual differences
at each step of a transformation script, making the optimization process more
transparent and manageable than manual code optimization. This approach aims
to produce highly optimized code that is easier to review, debug, and maintain,
and that can adapt as the intended program or target hardware evolves. OptiTrust
is capable of performing a wide range of transformations, including core data
layout transformations (e.g., array-of-structure to structure-of-array, array tiling),
instruction-level transformations (e.g., reordering, constant propagation,
pattern-based rewriting), and common control flow transformations (e.g., loop fission, fusion, unrolling, tiling).

The workflow with OptiTrust involves describing optimization steps in transformation scripts,
which are expressed as OCaml programs. The framework provides an internal
library for manipulating a simplified Abstract Syntax Tree (AST)
that handles a large subset of C99, OpenMP pragmas, C11 alignment directives,
and some C++ features. The original source code is encoded into this
OptiTrust AST and can be decoded back to C code. Developers use a user-level
library of general-purpose transformations, which can be extended with custom,
domain-specific optimizations by composing existing transformations or directly
manipulating the AST. An interactive interface allows executing specific lines
or blocks from the transformation script and visualizing the corresponding diffs.
OptiTrust uses OCaml for expressing transformation scripts and implementing the framework,
leveraging OCaml's type-system, native compilation, higher-order functions,
and pattern matching for robust and efficient script development.
The framework works with C/C++ languages, parsing them into its internal AST for transformation.

\subsection[Xevolver]{Xevolver~\parencite{xevolver_2014_0016}}

Xevolver, an XML-based code translation framework, was proposed by Hiroyuki Takizawa,
Shoichi Hirasawa, Yasuharu Hayashi, Ryusuke Egawa, and Hiroaki Kobayashi.
The paper introducing Xevolver was published in 2013.

Xevolver is an extensible programming framework designed to separate
platform-specific optimizations from application codes, particularly
within the domain of high-performance computing (HPC) application migration and evolution.
It enables programmers to define their own custom code translation rules,
which is crucial for handling the special demands of individual systems, compilers,
libraries, and applications . Unlike predefined directive-based transformations such
as OpenMP or OpenACC, Xevolver allows for user-defined code transformations that complement
standard approaches, preventing over-complication of codes while optimizing HPC applications.
The framework is especially useful for incrementally improving the performance
of existing applications without significant code modifications, thereby enhancing
performance portability across diverse platforms. Xevolver can handle various
code modifications, including those that involve inserting, removing,
or replacing code fragments based on specific patterns, which is a common need
in HPC optimization.

The workflow with Xevolver involves parsing an application program,
written in C or Fortran, into an Abstract Syntax Tree (AST). This AST is then exposed
as an XML document, referred to as an XML AST, allowing expert programmers
to manipulate it using XML-related technologies. The framework supports
user-defined code translation rules, or "translation recipes," which are
written in external files. For transformations, Xevolver primarily uses XSLT
(eXtensible Stylesheet Language Transformations) to describe these translation rules,
leveraging XML's capabilities for data manipulation. After the XML AST is modified
according to these rules, it is converted back to a ROSE AST and then unparsed
to generate a modified version of the original application program. This process
allows for systematic code transformations, such as loop interchange for numerical
simulation codes, which can be critical for adapting to different platforms like GPUs.

Xevolver works with application codes written in C and Fortran. The framework itself
utilizes XML as the language to describe transformations, specifically employing
XSLT for defining translation rules. This choice allows Xevolver to leverage mature
and standardized XML technologies, making it accessible for programmers to learn and use.
By using XML, Xevolver facilitates the separation of platform-specific optimizations
from the application code itself, contributing to better maintainability and performance portability.

\subsection[Loopy]{Loopy~\parencite{rival_loopy_2016_0008}}

Loopy is a Python-based tool developed by Andreas Kloeckner, primarily known for
his work on PyOpenCL and PyCUDA, which are Python wrappers for OpenCL and CUDA respectively.
Loopy's mission is to provide a versatile source-to-source transformation framework,
particularly for loop nests, to facilitate high-performance computing (HPC) optimizations.
It serves as a crucial component in projects requiring fine-grained control over code
generation and optimization, allowing researchers and developers to explore various
transformation strategies programmatically. Its design emphasizes flexibility and
extensibility, making it suitable for complex optimization tasks in scientific computing.

Loopy is capable of performing a wide array of source-to-source transformations
on array-based computational kernels, which are fundamental to HPC.
These capabilities include common loop optimizations such as loop tiling (blocking),
unrolling, fusion, and permutation. Beyond these standard transformations,
Loopy can also handle more advanced operations like software pipelining,
dependency-aware scheduling, and vectorization, enabling the generation of highly
optimized code for parallel architectures. It allows for the symbolic manipulation
of loop nests, providing a high degree of control over the generated code structure
and memory access patterns. This makes it particularly useful for optimizing kernels
for GPUs and other many-core processors where explicit control over parallelism
and memory hierarchy is critical for achieving peak performance.

The typical workflow with Loopy begins by representing a computational kernel
as a Loopy kernel object, which is essentially a symbolic description of the computation.
This kernel can be defined directly in Python using Loopy's API, or it can be
imported from existing code. Once the kernel is represented, users apply a sequence
of transformation instructions, also expressed in Python, to modify the kernel's structure.
These instructions describe how loops should be tiled, unrolled, reordered, or how
data should be mapped to memory. After applying the desired transformations, Loopy generates
optimized C or OpenCL C code, which can then be compiled and executed on the target hardware.
Loopy primarily works with Python for defining kernels and transformations, and it
generates C/OpenCL C for execution. The transformations themselves are described using Loopy's
domain-specific language (DSL) embedded within Python, allowing for powerful and programmatic manipulation of the kernel's structure.

\subsection[CHiLL]{CHiLL~\parencite{chen2008chill_0002}}

CHiLL (Composing High-Level Loop Transformations) is a framework developed by Chun Chen,
Jacqueline Chame, and Mary Hall, with the primary mission of providing a general and robust
system for loop transformations in compilers. It is a key component of a broader compiler
project focused on empirical optimization, where various optimized code variants are
evaluated through execution on target architectures.
This framework aims to bridge the performance gap between manually optimized and
compiler-generated codes, particularly for complex loop nests in scientific and engineering applications.

CHiLL is capable of supporting a broad collection of loop transformations, including permutation,
tiling, unroll-and-jam, data copying, iteration space splitting, fusion, and distribution.
Unlike many existing polyhedral transformation models, CHiLL goes beyond their limitations
by handling imperfect loop nests and non-rectangular loop bounds. The framework
emphasizes robustness and applicability, providing a uniform representation of iteration
spaces and statements to facilitate the composition of multiple transformations without
intermediate code generation. It also focuses on generating high-quality code by
minimizing overhead in the presence of complex code constructs.

The workflow with CHiLL involves taking the original code and a high-level
transformation script as input to generate an optimized code version.
The framework uses a polyhedral representation of iteration spaces, which allows
it to abstract original code constructs and manipulate iteration spaces, facilitating
the composition of transformations . Transformations are described using a high-level
transformation script interface, which specifies the sequence of transformations with
minimum required parameters, enabling the search for alternative variants and parameter
values. After applying the transformation algorithms, the final transformed code is
generated from the modified iteration spaces and statements. The framework leverages
the Omega Library for polyhedra manipulation and scanning, with improvements to
its functionality. While the document does not explicitly state the programming languages
CHiLL works with or uses for transformations, the context of Fortran loop transformation
and the description of a "script interface"  suggest it's designed to interact with
Fortran code and uses a script-based approach for defining transformations.

\subsection[ASSIST]{ASSIST~\parencite{lebras2017assist_0020}}

ASSIST (A Source-to-Source Transformation Tool for HPC Applications) is an open-source
framework developed by Youenn Lebras, Andres S. Charif Rubial, Romain Dolbeau,
and William Jalby, among others, with the primary goal of assisting application programmers
in achieving better productivity and code performance. It functions as a directive-oriented
source-to-source manipulation tool, integrated into the MAQAO toolset, and is designed
to leverage both static and dynamic profiling data to guide code transformations.
This integration allows ASSIST to take advantage of the comprehensive static and
dynamic profiling data produced by other MAQAO tools, such as LPROF for profiling
hotspots, CQA for code quality metrics (e.g., vectorization ratio), and VPROF for
function and loop value profiling.

ASSIST is capable of handling various types of transformations, ranging from common loop
transformations to more specialized ones. It supports common loop transformations
like interchange, unroll (including full unroll), strip mine, and tile. Beyond these,
ASSIST implements constant propagation and local dead code elimination to clean up
transformed code and eliminate useless chunks generated by specialization.
A key capability is specialization, which involves creating particular versions
of code by considering specific values of variables, such as specializing a loop
based on induction variable values or a function based on parameter values. The tool
also features block vectorization, which forces the compiler to vectorize loops
using SIMD directives, prevents peeling code, and adapts iteration counts to vector
length, proving effective when compilers fail to vectorize loops properly.

The workflow with ASSIST involves parsing the original source code into an Abstract Syntax Tree (AST)
using the ROSE compiler infrastructure . Transformations can be specified through two
alternative approaches: using directives or a Lua script. The script interface allows
for the complete hiding of transformations, enabling users to specify a sequence
of transformations with minimal parameters. After the AST is transformed according
to the directives or script, ASSIST generates a modified source file as output.
The tool is designed to work with C, C++, and Fortran languages, prioritizing scientific
applications in the HPC field . The use of the ROSE compiler infrastructure
is crucial as it is an open-source tool capable of manipulating the AST of C, C++,
and Fortran source codes.

\subsection[POET]{POET~\parencite{yi_poet_2012_0011}}

POET (Parameterized Source-to-source Program Transformations) is a scripting language
developed by Qing Yi, affiliated with Rice University and the University of Texas at San Antonio.
It was designed to apply advanced program transformations to code in various programming
languages and to facilitate the rapid creation of ad-hoc source-to-source translators
between these languages  . This research was supported by grants from the NSF and DOE.
POET's mission is to provide programmable control over compiler optimizations for
high-performance computing and to reduce the development cost of source-to-source program
transformation, ad-hoc language translation, and domain-specific code generation.
It has been utilized to support numerous compiler optimizations, such as loop interchange,
parallelization, blocking, fusion/fission, strength reduction, scalar replacement, and SSE
vectorization.

POET is highly capable, supporting a wide array of program transformations and code
generation tasks. It is an interpreted language that uses external syntax descriptions
to dynamically parse and unparse code in arbitrary programming languages, making it
language-neutral. This dynamic parsing capability allows it to handle various languages
like C, C++, Java, and Fortran, and even domain-specific languages. Transformations are
defined as `xform` routines, which can use control-flow, build compound data structures,
and invoke built-in operations like pattern matching and AST replacement.
POET also supports parameterization of transformations, allowing a single script to produce
diverse outputs based on command-line parameters, which is crucial for empirical
tuning of compiler optimizations  . The framework includes a robust optimization library
(`POET opt library`) containing many advanced compiler transformations that can be invoked
by scripts.

The workflow with POET typically involves using a POET script to define the transformation
process. The script specifies which input files to parse using particular syntax descriptions,
what transformations to apply, and which syntax to use for unparsing the result.
The original source code is parsed into an internal representation, an Abstract Syntax Tree (AST),
which can then be manipulated by \lstinline{xform} routines. POET employs a dynamic recursive descent
parsing algorithm that interprets the syntax of the input language at runtime, rather
than relying on statically defined parsers. It uses code templates to specify both the concrete
syntax and the internal representation of a language. After transformations,
the modified AST is unparsed back into a source file. POET is designed to work with C, C++, Java,
and Fortran. It uses its own scripting language, POET, to describe transformations.
The ROSE compiler infrastructure is leveraged by POET for analysis, particularly for
supporting programmable control of optimizations outside the compilers.

In summary, POET is a versatile, interpreted scripting language designed by Qing Yi
to enable flexible and programmable source-to-source transformations for high-performance
computing. It supports a wide range of languages and transformations through dynamic
parsing, AST manipulation, and a high-level scripting interface, often integrating
with tools like the ROSE compiler for advanced analysis.
